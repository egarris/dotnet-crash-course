# Module 2 — Boundaries, Contracts, and Concerns

Most discussions of "abstraction" in software collapse the word into something organizational — moving code into a separate file, putting it behind a function, choosing the right folder. That is the easy half of the idea. The harder half is the one this module is built around. A useful abstraction is a promise paired with a refusal: a promise about what callers can rely on, and a refusal to promise anything else. The first half is what makes the abstraction useful. The second half is what makes it durable.

This is the load-bearing module of the course. Almost every module that follows it is an instance of the ideas in this one. Dependency injection is one specific way of expressing a promise about how dependencies are obtained. Layered architecture is one specific answer to where promises and refusals belong. Repositories, mediators, and validators are not patterns to memorize — they are answers other engineers reached when they faced the same design question this module asks the reader to take seriously.

The reason to spend time on the question itself, rather than the patterns that answer it, is straightforward. Patterns work where they were originally designed to work. They mislead where they weren't. The skill that survives is the ability to look at a chunk of code and reason about whether the boundaries inside it are doing useful work — and if they're not, to know what useful work they could be doing instead.

## The promise and the refusal

The framing has a rigorous precursor in Bertrand Meyer's Design by Contract (*Object-Oriented Software Construction*, 1988), which formalized the idea as preconditions on the caller, postconditions on the implementation, and invariants the abstraction maintains across calls. The version this module uses is less formal and more accessible, but the lineage is worth knowing for anyone who wants to read further.

Treat the abstraction-as-promise idea concretely. Take any class, function, module, or service in front of you. Four questions are usually enough to know whether it is a real abstraction or a name with code under it.

What is it *for*? Not what does it contain — what does it accomplish for someone outside it? The job description should fit in a sentence. If it doesn't, the candidate is either two things glued together or something whose author hasn't yet decided what it is.

What does it *promise*? What can callers rely on? What is the contract that callers will write code against, and that the implementation will be held to even after it is rewritten? The contract is the stable surface; the implementation is everything underneath. If the contract isn't clearly distinguishable from the implementation, the abstraction has no boundary — only a name.

What does it *hide*? A useful abstraction conceals at least one decision that was likely to change. If nothing is hidden, the abstraction is doing no work that a single line of inlined code couldn't do. The harder question is whether what's hidden is the right thing to hide — whether the things free to change are the things that actually want to change in this codebase, in this team, on this timescale.

What does it *cost*? Every abstraction is paid for. Some of the cost is cognitive: an engineer who needs to change something on either side of the boundary has to learn what the boundary is hiding before they can act. Some is runtime: indirection, virtual dispatch, allocation that wouldn't exist without the boundary. Some is evolutionary: a boundary drawn well today can constrain change in a year, when the assumptions that justified it no longer hold. The skill that distinguishes mature design from earnest design is being honest about these costs — not to refuse abstractions, but to demand that each one earn its keep against a known bill.

These four questions have a corollary that makes them sharper. If the answer to "what is this for" runs more than a sentence, or the answer to "what does it promise" is "everything its implementation does," or the answer to "what does it hide" is "nothing important," or the answer to "what does it cost" is "nothing the team has noticed," then the candidate is either ceremony or an unbilled cost. The work of refactoring isn't to move code somewhere else; it's to find the answers that aren't yet present in the design.

## What's worth hiding

The idea that abstraction is fundamentally about *hiding* is older than most active software engineers. David Parnas argued in 1972 that the unit of modularization should be a design decision likely to change, and that the rest of the system should be insulated from that decision. The shape of the argument has held up because the underlying observation has held up. Software lives in a world where requirements, environments, and dependencies change, and the systems that absorb changes well are the ones whose decisions live behind boundaries that hide them.

That phrasing is precise enough to use as a working test. Look at a candidate boundary and ask: which decisions live behind it that callers don't see? If a future engineer changes the database from one provider to another, which boundaries do they cross? If an HTTP API gets a new authentication scheme, what code knows about the old one? If the validation library gets swapped, how many files compile differently?

The answer to those questions is the design's actual modularity. Most systems claim more modularity than they have. A repository named `OrderRepository` that returns the persistence framework's tracked entity graph hides almost nothing — when the persistence layer changes, the change ripples through every caller. A repository that returns purpose-shaped result objects hides the persistence framework entirely; the change can be absorbed.

Two well-worn critiques of the hiding discipline are worth taking seriously. The first is that information hiding can be over-applied. When every decision lives behind a boundary, understanding the system requires crossing every boundary, and indirection grows faster than insight. John Ousterhout's *A Philosophy of Software Design* makes the case that the goal is *deep* modules — abstractions that hide a substantial amount of complexity behind a small interface — and that *shallow* modules, ones whose interface is nearly as large as their contents, can be worse than no abstraction at all. The principle of hiding is right; applying it well is harder than the principle makes it sound.

The second critique is that predicting which decisions will change is unreliable. Engineers consistently misjudge it. Decisions believed to be stable get revised; decisions hidden behind elaborate boundaries turn out to have been static all along. The skill of reading a codebase well enough to concentrate hiding where it pays off is real, but it is also fallible, and the wrong predictions are paid for in the same indirection cost as the right ones. A team that designs as if it has perfect foresight produces over-engineered systems; a team that designs as if foresight is impossible produces under-abstracted ones. The middle path is to make boundary decisions reversible enough that getting them wrong is recoverable, and to revisit them when the codebase reveals the predictions were wrong.

## Dependency direction

Once an abstraction is a promise paired with a refusal, the direction of dependencies follows. Code that calls an abstraction depends on the promise — the contract. Code that implements an abstraction is constrained by the contract. The contract is the shared object; both sides depend on it. The implementation does not get to decide what the contract is, and the caller does not get to reach past the contract into the implementation.

That sentence is what "depend on abstractions, not concretions" is gesturing at. When a caller depends on an interface, it depends on the smaller, more stable contract; the implementation can change underneath without breaking the caller. When a caller depends on a concrete class, it depends on everything that class does, including the parts that should have been free to change.

The same logic scales up to layers. A domain layer (the part of the codebase that expresses the business problem in business terms) depends on its own abstractions of persistence and external services; the persistence and external service layers depend on the domain by implementing what the domain promised. The dependency arrows point toward whichever party owns the contract. When they point the other way, the design has failed at exactly the question this module asks: it has decided that the higher-level concept is constrained by the lower-level implementation, instead of the other way around.

The signal that dependencies are pointing the wrong way is usually concrete. Domain code that imports the database driver. A service interface whose method signatures are shaped around HTTP. A validation rule that knows about the JSON serializer. Each of these is a decision that the abstraction was supposed to hide leaking back into the code that the abstraction was supposed to insulate.

The framing in this section follows the tradition of Robert Martin's Dependency Inversion Principle and the broader Clean Architecture / Hexagonal / Ports-and-Adapters family. The reasons behind that tradition are sound — domain logic is what the system is built around; persistence and protocols are details that should be free to change underneath it. The tradition is not the only one. Some successful systems point dependencies the "wrong" way deliberately, allowing higher-level concepts to depend on stable lower-level services directly when the indirection added to "fix" the dependency direction would be overhead with no offsetting benefit. The discipline matters most where the lower-level concept is volatile and the higher-level concept is the part the system is organized around. The discipline matters less when the lower-level concept is stable infrastructure that everyone is content to depend on directly. Deciding which case applies is the same skill as deciding what is worth hiding — and it is the skill, not the rule, that distinguishes durable architecture from cargo-culted layering.

This module addresses *tactical* boundaries: where to draw lines within a service or codebase, between classes, between layers. The *strategic* question — where to draw the boundaries of services and bounded contexts in a larger system — is a different exercise of the same skill, and Module 17 takes it seriously. The reader should know the strategic question exists one level up; this module is asking it one level down.

## Coupling, cohesion, and one reason to change

Two older terms organize the same observation from a different angle. Coupling describes how much two pieces of code know about each other. Cohesion describes how much the parts of a single piece of code belong together. The target most engineers learn — high cohesion, low coupling — is right but vague. The reason it is right becomes clearer once you read it through the abstraction-as-promise frame.

High cohesion means the contents of an abstraction share a job. They change for the same reason. They serve the same caller. They have the same audience. Low cohesion means the abstraction is a junk drawer — its contents are there because they fit nowhere better, not because they belong together. Junk-drawer abstractions are hard to name (because there isn't a single thing to name), hard to test (because their concerns interact unpredictably), and hard to change (because each change affects unrelated callers).

Low coupling means the contract between two abstractions is small, stable, and explicit. High coupling means the boundary leaks — callers depend on more than the contract says, or implementations promise more than the contract allows. High-coupling boundaries break under pressure: when one side changes, the other side has to change with it, even when nothing about the contract was supposed to change.

The most useful re-reading of the Single Responsibility Principle is in this language. The principle is often quoted as "do one thing," which is wrong in a way that matters: many useful abstractions do several things, all of which serve the same purpose. The better reading, attributable to Robert Martin, is that an abstraction should have one reason to change. That reason is the audience it serves, or the decision it hides, or the job it does. When two unrelated reasons to change live in the same abstraction, the abstraction will eventually be split — usually by an engineer who is trying to make one change without making the other and discovers they cannot.

A more recent challenge to the textbook reading is worth naming. Vlad Khononov's *Balancing Coupling in Software Design* (2024) argues that "high cohesion, low coupling" is too binary to capture how real systems work. Some genuinely multi-faceted domains resist clean cohesion. Some pairs of components benefit from being closely coupled because they change together for legitimate reasons, and inserting indirection between them adds cost without insulating anything. The principle is still useful as a default; the application requires the same kind of judgment as deciding what is worth hiding. A reader who finds the textbook answer feeling wrong in front of a particular piece of code may be looking at one of the cases the textbook answer doesn't fit, rather than at code that needs to be reshaped to match the textbook.

## When refusal isn't enough

A complication runs through everything above that is worth carrying forward. Hyrum's Law, articulated by Hyrum Wright, observes that with a sufficient number of users of an interface, it does not matter what the contract promises: every observable behavior of the system will eventually be depended on by some caller. The contract is what was promised; the *behavior* is what callers actually rely on, and behavior includes things the contract refused to commit to. A method that returns items in insertion order without promising the order will eventually have a caller who relies on the order and breaks when it changes. A timing characteristic that the contract didn't mention will be load-bearing in some downstream system that observed it and assumed it. Mature platforms narrow their public surface area aggressively for this reason — every exposed method, every observable side effect, every reproducible quirk becomes a commitment under enough use, regardless of what the documentation says.

The gap has a deeper structure worth naming. Wilfrid Sellars distinguished between the *space of reasons* (the normative space where we make commitments, give justifications, and hold each other to what we said we would do) and the *space of causes* (the descriptive space where things happen as a matter of fact). A contract lives in the space of reasons: it is what was committed to. Behavior lives in the space of causes: it is what the system does. Hyrum's Law is the observation that callers, working in their own code, can only directly see the space of causes; they form load-bearing dependencies on what they observe, not on what was promised. The two spaces are categorically different — the contract isn't an under-specified description of behavior, and behavior isn't an empirical record of the contract — and that difference is why the gap between them is structural rather than merely practical. The same shape of problem appears anywhere a normative commitment meets a causal reality: specs and implementations, intent and telemetry, design documents and what production does.

The implication for the four questions is small but important. The discipline of refusing to promise what you don't intend to commit to is what keeps the gap between contract and observed behavior small enough to manage. The discipline doesn't close the gap; it bounds it. Closing the gap would require the willingness to actively change observable behavior on a regular cadence, so that callers can't form load-bearing dependencies on it — itself a design discipline most teams don't have the appetite for. "What does it promise?" is therefore best read in the long run as "what does it commit to, including the parts of its behavior that are observable enough that callers will depend on them whether you intend it or not?" That broader reading is what makes the refusal half operational rather than aspirational.

## OOP as the toolkit for expressing boundaries

The language features the reader saw in Module 1 — classes, interfaces, abstract types, generics, value semantics, sealed and virtual modifiers — are the tools .NET gives you for expressing boundaries. Each one corresponds to a design move worth naming.

A class encapsulates state and the operations on it. Its public surface is the contract; its private fields and methods are what's hidden. Constructors are where invariants get established — by the time the constructor returns, the object is supposed to be in a valid state, and callers are supposed to be able to rely on that. The class as a whole is one of the most common units of "an abstraction" in C# code, and it earns that status when its public surface is small relative to its internals.

An interface separates the contract from any particular implementation. It is the most direct way to express "this is the promise; the implementation is somewhere else." Code that takes an interface as a constructor parameter depends on the contract, not the implementation, which is why interfaces are the natural tool when an abstraction needs more than one implementation, when it needs to be substitutable for testing, or when its implementation is genuinely likely to change. Interfaces are also commonly overused — declared for every class out of habit rather than because the contract-implementation distinction earns its keep. An interface with exactly one implementation, no test substitution, and no foreseeable second implementation is paying the indirection cost without buying the flexibility.

Abstract classes occupy a middle ground. They share concrete code with their implementations and require subclasses to fill in particular methods. They are useful when several implementations share genuine behavior; they are misused when they are used as a way to share unrelated code by inheritance. The longer-running design tradition prefers composition (assembling behavior from cooperating objects) over inheritance (sharing behavior through a class hierarchy) for exactly this reason: composition lets each piece of shared behavior be its own abstraction with its own contract, while inheritance bundles them together with no opt-out. Abstract classes remain the right tool when the shared behavior genuinely belongs together; composition is the right tool more often than someone who learned object orientation through inheritance trees first realizes.

Sealed and virtual modifiers are the language's way of saying which boundaries are stable. A sealed class refuses to be extended; a virtual method invites override. Marking a class sealed is a refusal to promise that subclasses are supported; leaving methods non-virtual is a refusal to promise that their behavior can be redefined. These are design choices about what the abstraction promises and what it leaves free.

Generics let an abstraction make a promise that holds for many types. A list's contract is the same regardless of what it contains; the contract is parameterized by the element type. Generic constraints let the contract be more specific without losing reusability. Variance — the `in` and `out` modifiers on generic type parameters — extends the promise across related types: an `IEnumerable<Cat>` can be assigned to an `IEnumerable<Animal>` because the contract only produces values, and an `IComparer<Animal>` can be used where an `IComparer<Cat>` is expected because the contract only consumes them. Used well, generics let the type system carry information that would otherwise have to live in documentation or in casts. Used poorly, generics paper over abstractions that don't have a coherent shape — the type parameter is doing the work of distinguishing concerns that should have been distinguished by separate types.

Value-type semantics (structs, records, immutable objects) let the abstraction commit to its own equality and lifetime. A reference type's identity is its memory address; two instances are equal only if they refer to the same object. A value type's identity is its contents; two instances with the same fields are interchangeable. Choosing the right semantics is a design decision about what the abstraction means: a money amount, a coordinate, a color — these are values, not entities, and modeling them as such forces the codebase to treat them consistently.

None of these tools are mandatory. They are vocabulary. The point is that good designs in C# tend to make their boundary choices visible in the language, where the compiler can help enforce them, rather than burying them in convention or documentation.

## Naming as the forcing function

Names carry compressed claims about purpose. A useful abstraction has a name that fits the answer to "what is it for?" without strain. A junk-drawer abstraction has a name that strains — `OrderManager`, `UserHelper`, `DataService`. Names ending in *-Manager*, *-Helper*, *-Util*, or *-Service* are not always wrong, but they correlate strongly with abstractions that have not yet decided what they are. The author has reached for a generic suffix because no specific noun fits.

This is one reason naming is sometimes the fastest way to find a design problem. When a class's contents resist a clean name, the contents probably resist a clean abstraction. When a method's behavior cannot be summarized in five words, the method is probably doing two things. The exercise of writing one-line job descriptions for every class in a service often surfaces more design issues per minute than any other refactoring activity.

The corollary cuts both ways. A name that fits cleanly does not guarantee a good abstraction; the abstraction can still be cohesive and well-named while pointing the wrong way in the dependency graph. But a name that does not fit cleanly is reliable evidence that something underneath is wrong.

## A production failure that lives in the boundary

Here is a real-shaped scenario, anonymized. A service in a mature codebase is built on a generic base class meant to remove repetition: every entity type gets a base repository that handles persistence. The base repository uses reflection to find primary keys, accepts string arguments for related-data inclusions, and returns the persistence framework's tracked entity graphs (the framework's representation of database rows as connected objects, with modification tracking) to its callers. Domain services receive these tracked entities and pass them to mappers; the mappers build response objects; the response objects are returned to the API.

The team treats this as a layered architecture. There is a repository layer, a service layer, a controller layer. Code is in the right files. Files are in the right folders. By every organizational measure, the architecture is in place.

Then a database refactor lands. A column changes type. A relationship changes cardinality. The team expects the repository to absorb the change. It doesn't. The change ripples through services, through mappers, through controller signatures, into the API contract that customers depend on. A change that was supposed to be local to persistence is felt by every caller in the stack.

The problem isn't that the team forgot a layer. The problem is that the layers were named but not load-bearing. Each layer passed the persistence framework's data structures through to the next; nothing was hidden. The repository made no promise about its return shape that wasn't dictated by the database. The service made no promise about its arguments that wasn't dictated by what the repository returned. The dependencies pointed the wrong way: the API's shape was determined by the schema, not the other way around.

The fix wasn't to add more layers. The fix was to make the existing layers do the work their names had been claiming. The repository started returning purpose-shaped result objects, projecting only the fields each caller actually used. The services took inputs in the shape their domain logic needed, not in the shape the database returned. The boundaries became real because they started hiding something. After the next schema change, the rippling stopped at the repository.

The fix was not free. Purpose-shaped result objects mean more types to maintain, more projections to keep in sync, and slow divergence over time as different callers' needs drift apart. This is approximately the Read Model pattern from CQRS (Command-Query Responsibility Segregation — a design where the shapes used for reading data are deliberately separated from the shapes used for writing it), and it carries the burden CQRS carries: a codebase with twice as many shapes, asking the team to keep both halves coherent. In return, schema changes stop rippling through the API, persistence concerns stay out of domain code, and the test surface for each layer narrows to the contract that layer actually owns. That trade is usually worth it on long-lived services with multiple clients. It usually isn't on short-lived services with one. The discipline is a balance, not a free win.

The strongest counterargument to the discipline as a whole is worth taking seriously. Some teams genuinely do not need it — small services with one client, short lifespans, schemas that change in lockstep with the API. In those cases, the overhead of hiding what isn't going to be hidden meaningfully is real, and writing a thin pass-through is the right choice. The discipline is useful where it pays for itself and dead weight where it doesn't. The skill is reading the project well enough to know which case you are in. The mistake the team in the scenario made was not that they used a layered structure — it was that they used a layered structure on a long-lived service with multiple clients while refusing to make the layers do any actual work.

One last note on this failure mode: the pattern often persists because the test suite quietly sustains it. When tests mock tracked entities and pass them through the test surface as if they were the contract, they validate that the broken design "works." Tests that stop at the wrong layer are a co-conspirator in boundaries that don't hold, and the repair work in the scenario above included rewriting the tests against the new layer contracts. Module 18 picks this thread up properly.

## A load-bearing point

The first place this module's ideas get exercised under pressure is the design review. Someone proposes a structure. Someone else asks why the boundary is there. The defensible answers are versions of the four questions: it is there because it has a job the surrounding code does not have, because it has a contract callers can rely on, because it hides a decision that this codebase, on this team, on this timescale, has reasons to keep changeable, and because the cost it imposes is justified by the work it does.

The answers that don't hold up are the ones that come from organizational habit. "Because we always have a service layer." "Because the framework expects it." "Because the architecture diagram has one." These are not wrong on their face; they may even be locally correct. But they don't tell the room what work the boundary is doing, which means they don't tell the room when the boundary should change. A team that can defend its boundaries on the merits can move them when conditions change. A team that has only inherited reasons cannot.

The same skill is what makes AI-generated pull requests reviewable. AI tooling produces code that follows familiar shapes — the right number of layers, the right naming patterns, the right pieces in the right files. Whether the boundaries inside that code are doing useful work is a separate question, and it is the reviewer's question. A reviewer who can ask the four questions of any class that lands in a diff is doing work that the producer of the diff could not do.

## Closing

The shortest version of this module is that an abstraction is a contract about what gets to change, and the design discipline is choosing which decisions are worth that protection. The longer version is the rest of the course, where the same discipline reappears at every level — between functions, between layers, between services, between teams.

The question worth carrying forward into the rest of the foundation arc: when the next layer of code lands in front of you, can you state in one sentence what it promises, what it refuses to, and what it costs? If not, you are looking at a design that has not yet finished deciding what it is.
